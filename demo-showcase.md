## Demo Flow (5 steps, ~10 minutes)

### Step 1: Show the Cluster Architecture (1 min)

> **Talk track:** "We have a 3-node bare-metal Talos Linux cluster — 1 control plane + 2 workers, all AMD EPYC with SEV-SNP enabled."

```bash
# Show nodes
kubectl --kubeconfig talos-amd/kubeconfig get nodes -o wide

# Show the CoCo extension is installed on every node
talosctl --talosconfig talos-amd/talosconfig \
  -n 192.168.200.2,192.168.200.3,192.168.200.4 \
  -e 192.168.200.2 get extensions

# Show SEV-SNP is enabled in the kernel on all nodes
for ip in 192.168.200.2 192.168.200.3 192.168.200.4; do
  echo "=== $ip ==="
  talosctl --talosconfig talos-amd/talosconfig -n $ip -e $ip dmesg 2>&1 | grep "SEV-SNP enabled"
done
```

### Step 2: Show RuntimeClasses and Labels (1 min)

> **Talk track:** "We registered 3 runtime handlers. The key one is kata-qemu-snp — it uses a patched QEMU that implements the full SNP launch protocol with memory encryption and attestation."

```bash
# Show available runtime classes
kubectl --kubeconfig talos-amd/kubeconfig get runtimeclass

# Show node labels that drive scheduling
kubectl --kubeconfig talos-amd/kubeconfig get nodes \
  -o custom-columns='NAME:.metadata.name,SNP:.metadata.labels.coco\.confidentialcontainers\.org/snp,KATA:.metadata.labels.katacontainers\.io/kata-runtime'
```

### Step 3: Deploy a Confidential Pod (2 min)

> **Talk track:** "Now I'll deploy an Ubuntu pod using the kata-qemu-snp runtime. This launches a real QEMU virtual machine with SEV-SNP encryption — the host cannot read the guest's memory."

```bash
# Deploy on node2 (a worker node)
kubectl --kubeconfig talos-amd/kubeconfig apply -f - <<'EOF'
apiVersion: v1
kind: Pod
metadata:
  name: demo-confidential
  annotations:
    io.containerd.cri.v1.images/unpack: "false"
spec:
  runtimeClassName: kata-qemu-snp
  nodeName: node2
  containers:
  - name: ubuntu
    image: ubuntu:24.04
    command: ["/bin/sleep", "infinity"]
EOF

# Wait for it to start (~2 minutes for guest-pull)
kubectl --kubeconfig talos-amd/kubeconfig get pod demo-confidential -w
```

### Step 4: Verify the VM is Running with SEV-SNP (2 min)

> **Talk track:** "Let's exec into the pod and confirm this is running inside an SEV-SNP encrypted VM, not a regular container."

```bash
kubectl --kubeconfig talos-amd/kubeconfig exec -it demo-confidential -- bash
```

Inside the pod:
```bash
# Show kernel — this is the Kata guest kernel, not the host
uname -a

# Show SEV-SNP is active in the guest
dmesg | grep -i sev

# Key lines to point out:
#   "Memory Encryption Features active: AMD SEV SEV-ES SEV-SNP"
#   "SEV: SNP running at VMPL0"
```

### Step 5: Run Live Attestation (3 min) ⭐

> **Talk track:** "This is the key part — cryptographic attestation. We'll ask the AMD hardware to prove this VM is genuinely running in an SEV-SNP enclave, then verify the proof against AMD's certificate chain."

Still inside the pod:
```bash
# Create the /dev/sev-guest device
SNP_MAJOR=$(cat /sys/devices/virtual/misc/sev-guest/dev | awk -F: '{print $1}')
SNP_MINOR=$(cat /sys/devices/virtual/misc/sev-guest/dev | awk -F: '{print $2}')
mknod -m 600 /dev/sev-guest c "${SNP_MAJOR}" "${SNP_MINOR}"

# Download the attestation tool
apt-get update -qq && apt-get install -y -qq curl ca-certificates
curl -sLO https://github.com/virtee/snpguest/releases/download/v0.10.0/snpguest
chmod +x snpguest

# 1. Generate attestation report (hardware-signed)
./snpguest report report.bin request-file.txt --random

# 2. Fetch AMD's certificate chain
./snpguest fetch ca -r report.bin pem ./
./snpguest fetch vcek pem ./ ./report.bin

# 3. Verify the certificate chain
./snpguest verify certs ./
# Expected: ARK self-signed → ASK signed by ARK → VCEK signed by ASK

# 4. Verify the attestation report (the money shot!)
./snpguest verify attestation ./ ./report.bin
# Expected: "VEK signed the Attestation Report!"
```

> **Talk track (while verifying):**
> - "The attestation report is generated by the AMD hardware, not software — it can't be faked"
> - "The certificate chain goes all the way to AMD's root key (ARK)"
> - "VCEK is the chip-specific key — unique to this physical CPU"
> - "'VEK signed the Attestation Report' means we have cryptographic proof this VM is running in a genuine SEV-SNP enclave on verified AMD hardware"

### Cleanup

```bash
exit  # exit the pod
kubectl --kubeconfig talos-amd/kubeconfig delete pod demo-confidential --force
```

## FAQ

**Q: Does this add latency?**
A: VM startup adds ~2 min (guest image pull). Runtime performance impact is minimal (~2-5%) since SEV-SNP uses AES hardware acceleration.

**Q: Can the host admin read the pod's memory?**
A: No. SEV-SNP encrypts all guest memory with a key managed by the AMD security processor. Not even root on the host can access it.

**Q: What happens if someone tampers with the VM?**
A: The attestation report would fail verification — the measurement wouldn't match the expected state.

**Q: Is this the same as what the CoCo Helm charts do on Ubuntu?**
A: Yes — same Kata Containers, same QEMU, same attestation. We just packaged it as a Talos system extension instead of a DaemonSet.
